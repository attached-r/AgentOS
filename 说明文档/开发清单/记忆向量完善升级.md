 ---
  真正的 RAG + 记忆检索系统方案（V3 向量升级）

  核心架构变化

  当前（V2）：
  Runtime → Backend API → PostgreSQL ILIKE（关键词匹配）

  目标（V3）：
  Runtime → Qdrant（向量语义搜索）            ← 检索主路径
          → Backend API → PostgreSQL           ← 仅用于元数据管理（文档记录、审计）

  Qdrant 语义搜索替换掉所有 ILIKE，PostgreSQL 退为元数据存储。

  ---
  关键设计决策

  1. Runtime 直连 Qdrant，不走后端 — 减少一跳延迟，embedding + vector search 都在 Python 进程内完成。后端仅用于
  knowledge_doc / agent_memory 表的元数据管理。
  2. SemanticMemory 的 Neo4j 先缓后实 — 图存储先用内存 dict 兜底（_entities / _relations），Neo4j
  集成作为可选增强，不影响核心语义检索功能。
  3. embedding 模型选择中文优化版 — 默认 BAAI/bge-small-zh-v1.5（384
  维，中文优化），兼顾检索质量和性能。如需英文为主可切 all-MiniLM-L6-v2。
  4. Qdrant 本地优先，云端可选 — 开发期用 docker 跑本地 Qdrant，生产环境可切 Qdrant Cloud。QdrantVectorStore
  通过构造参数自动判断连接模式。
  5. 降级路径保留 — Qdrant 不可用时，RAGPipeline 和 EpisodicMemory 回退到现有 ILIKE 后端路径，不阻断主流程。
